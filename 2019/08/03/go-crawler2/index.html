go实现并发版爬虫

发表于 2019-08-03 | 分类于 go项目 ， 网络爬虫 | 阅读数 次

Introduction

在上一篇文章中,我们用go语言实现了一个单任务版爬虫，但是单任务版爬虫我们可以看到它爬取速度非常慢，每个页面都要去请求并且等待它返回以后才可以去请求下一个页面，这些动作完全可以并发地执行的，今天我们就来用go语言独特的goroutine实现并发吧！

并发版爬虫架构

单任务版爬虫架构

首先回顾一下单任务版爬虫的架构，首先Engine获得request并放到任务队列里，然后Engine不断从任务队列里去拿到下一个request，首先把request中的URL送给Fetcher，Fetch会根据URL从网上把URL的文本拿下来并且返回，然后再把这个文本送个Parser，Parser可以从文本中解析出有价值的信息，有价值的信息包括两部分：requests，items，requests就是新的request，engine拿到这些新的request再加进任务队列，items目前只是先打印出来，后续将会存储到数据库中。

并发版爬虫架构

现在我们嫌它太慢，要将某些部分并发，那哪里做并发呢！肯定要从耗时最长，等待时间最久的那个模块下手，那谁是耗时最长等待时间最多的模块呢，答案是Fetcher！因为Fetcher拿到URL要去互联网上获取文本，这里有一个远程的传输问题，所以Fetch是最慢的。而Parser是对本地内存文本的解析，这要比从网络上获取数据要快很多。

但是我们可以将fetcher和Parser合成一个较大的模块，我们可以看到Fetcher的输出就是Parser的输入，我们完全可以把Fetcher和Parser合成一个模块，我们把这个模块叫做Worker吧！

这就是Worker的代码，只是将Parser和Fetcher做了个整合，简单吧！

我们并发版爬虫就是要将Worker并发，需要开很多个Worker，我们将Worker并发了以后就将面临一个多对多的问题，有很多的Worker，就需要很多的request，很多的Worker就等待着要做它们，因此我们还需要建立一个模块(叫做Scheduler)负责调度任务，Scheduler会从engine得到request，然后将request放到任务队列中并且分发给Worker。

在单任务版爬虫里输入和输出都是函数的参数和返回值，但是在并发版里输入和输出就不是函数的参数和返回值了，那是什么呢？没错，那就是Channel！engine和Scheduler都是一个goroutine，Worker是很多个goroutine，这些goroutine通过channel互相连接，接下来我们就一一实现它们吧！

首先Scheduler要先实现Scheduler接口，它还要Submit,这里面有个循环等待的问题，我们只需要将其中一部分释放出来就可以解决了，解决的方法是在Submit中把向channel中放数据改用goroutine做，这样就可以有很多goroutine就不用循环等待了！

实现储存模块

首先我们要考虑有什么去存储爬取的数据呢？ElasticSearch是一个很好的选择：

  • 全文搜索引擎
  • 快速的存储，搜索和分析海量数据

首先ElasticSearch对我们这个项目能做什么呢？

  • 存储我们爬取的数据
  • 不需要建表，配置字段
  • json格式的文档
  • 全文搜索

ElasticSearch需要依赖java8，我们安装ElasticSearch还需要安装java8，这里我们用Docker去安装启动ElasticSearch

Docker是什么呢，这里简单介绍一下：

  • 容器引擎
  • 打包并发布应用程序，包里面包括系统，环境，配置，依赖
  • 沙箱机制
